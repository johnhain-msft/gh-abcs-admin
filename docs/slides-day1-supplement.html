Policy Inheritance

Enterprise → Organization → Repository cascade

┌─────────────────────────────┐ │ ENTERPRISE (top-level) │ Sets the floor │ Enforce · No Policy · Disable│ └──────────────┬──────────────┘ │ inherits ▼ ┌──────────────┴──────────────┐ │ ORGANIZATION │ Can add restrictions, │ Enabled · Disabled · Unconfigured │ never relax └──────────────┬──────────────┘ │ inherits ▼ ┌──────────────┴──────────────┐ │ REPOSITORY │ Content exclusions, │ (+ Individual prefs) │ personal settings └─────────────────────────────┘
  • Enforced — mandatory baseline; orgs cannot change
  • No Policy / Allowed — delegated; org owners decide
  • Disabled — blocked enterprise-wide; orgs cannot enable
Key principle: child entities can only ADD restrictions, never relax ones from above. Absence of a restrictive policy ≠ explicit permission. Enforced policies create a "compliance floor" that prevents policy drift. The decision tree short-circuits: if enterprise denies, org/repo are never evaluated.

Copilot Organization Policies

Key levers for org owners (subject to enterprise constraints)

Feature Policies

  • Access policy — who gets Copilot: all members or selected
  • Code completions — IDE inline suggestions
  • Copilot Chat — IDE, GitHub.com, Mobile, CLI
  • Code review — AI-assisted PR reviews

Security-Sensitive Policies

  • Public code matching — block suggestions matching public code
  • Cloud agent — autonomous code changes (evaluate carefully)
  • Agent mode (IDE) — multi-step agentic workflows
  • MCP servers — external integrations (security review first)
⚠️ Security-by-default: Block public code matching, disable cloud agent & MCP servers until reviewed. Delegate agent mode to org owners.
Each policy has 3 states at org level: Enabled, Disabled, Unconfigured. Unconfigured = disabled until explicitly set. Best practice: explicitly configure ALL policies rather than leaving them unconfigured. Privacy policies: block prompt/suggestion collection to maintain data privacy (code is not retained). Preview features should be disabled in production.

Content Exclusions

Prevent Copilot from accessing sensitive files

How It Works

  • Glob patterns exclude files from AI context
  • Inline suggestions disabled for excluded files
  • Chat cannot reference excluded content
  • Configurable at repo, org, and enterprise level

Common Patterns

  • **/.env, **/.env.*
  • **/secrets/**, **/*credential*
  • **/*.pem, **/*.key
  • **/terraform.tfvars, **/*.tfstate

Key Details

  • Propagation delay: ~30 minutes (or reload IDE)
  • Enterprise exclusions apply to all users regardless of licensing org
  • Org exclusions apply only to users licensed through that org
⚠️ Exclusions do NOT apply to: Copilot CLI, coding agent, or agent mode. Disable those features if exclusion is critical for compliance.
After configuring exclusions, wait ~30 min or reload the IDE. To test: open an excluded file — no inline suggestions should appear, and Copilot Chat should not be able to use the file content. Enterprise-level exclusions are the strongest: they apply to ALL Copilot users in the enterprise. The gap with CLI/agent mode/coding agent is a critical security consideration — call it out explicitly.

Audit Log Overview

What's Captured

  • Settings changes & access control
  • User membership & role changes
  • Repository management events
  • App permissions & webhooks
  • Git events (push, clone, fetch)
  • Authentication & billing events

Retention & Access

TypeRetentionAccess
Web events180 daysUI, API, streaming, export
Git events7 daysAPI only, streaming
StreamedIndefiniteYour external storage
⚠️ Git events have only 7-day retention — always enable streaming for long-term capture.
Enterprise audit log aggregates across ALL child orgs — it's the primary tool for enterprise-wide monitoring. Org audit log is more focused for org admins without enterprise access. Default UI shows 3 months; use the `created` qualifier to access up to 180 days. Enterprise owners must explicitly enable IP address disclosure — it's off by default. EMU enterprises get additional user security log events in the enterprise log.

Searching the Audit Log

Phrase qualifiers for precise filtering

Key Qualifiers

QualifierExample
action:action:repo.create
actor:actor:octocat
created:created:>=2025-01-01
repo:repo:my-org/my-repo
operation:operation:authentication
country:country:US

Example Queries

  • Repo creation by user in last 30 days:
    action:repo.create actor:octocat created:>=2025-05-01
  • Branch protection changes:
    action:protected_branch.update created:>=2025-04-01
  • Member removals:
    action:org.remove_member created:>=2025-01-01
Multiple qualifiers are AND'd together. Date ranges: created:2025-01-01..2025-03-31
Qualifiers are space-separated and AND'd. The `created` qualifier is crucial — default UI only shows 3 months, but events exist for 180 days. Export options: JSON or CSV from UI for web events, compressed JSON for Git events. API rate limit: 1,750 queries/hr per user+IP. GraphQL supports org-level audit log queries with ~90-120 days of data.

Why Streaming Matters

  • Overcomes 180-day / 7-day retention limits
  • Required for SOC 2, HIPAA, PCI-DSS compliance
  • Enables real-time alerting in SIEM tools
  • Supports up to 2 endpoints simultaneously

Buffering Rules

  • Paused < 7 days → no data loss
  • Paused 7–21 days → resumes from 1 week back
  • Paused > 3 weeks → resets to current time
  • Datadog: max 18-hour pause window
Stream format: compressed JSON (.json.gz), at-least-once delivery, deduplicate via `_document_id`. Health checks run every 24 hours — you get 6 days to fix a misconfigured stream before events are dropped. Multi-endpoint (preview): stream to S3 for storage AND Splunk for alerting simultaneously. Azure Event Hubs is ideal for Microsoft Sentinel integration. Not supported in Azure Government (Blob + Event Hubs).

Dormant Users

30 days without qualifying activity = dormant

Qualifying Activities ✅

  • SAML SSO authentication
  • Creating/deleting repositories
  • Push to internal repo via HTTPS
  • Creating/closing issues & PRs
  • PR review comments
  • Starring a repository
  • Joining an organization

NOT Qualifying ❌

  • PAT-based access (CI/CD pipelines!)
  • SSH key-based Git operations
  • GitHub App-based workflows
  • Git ops on private repos
  • Reading/browsing without interaction
  • Public repo activity outside enterprise
⚠️ #1 gotcha: devs using PAT/SSH daily may appear dormant. Cross-reference audit log before reclaiming!
The 30-day window is fixed — cannot be configured. You cannot manually mark a user as active; they must perform a qualifying activity. Download the dormant users report from Enterprise → Compliance → Reports. Also viewable in Enterprise → People → filter "Dormant". License reclamation workflow: 1) Identify via report, 2) Cross-reference with audit log for PAT/SSH activity, 3) Confirm with managers, 4) Remove via UI or `removeEnterpriseMember` GraphQL mutation, 5) Clean up IdP access. For EMU: soft-deprovision via IdP (reversible) to free the license.

Section 3 · ~15 min

Cost Centers Deep Dive

Departmental chargeback, tracking, and budget governance

This section covers the billing feature most requested by finance teams — cost centers. Cost centers let enterprises allocate GitHub spending across business units, projects, or departments for internal chargeback and budget accountability. We'll cover design patterns, allocation rules, budget alerts, Azure integration, and automation.

What Are Cost Centers?

Logical groupings for billing allocation and chargeback

  • Purpose — allocate GitHub costs to departments, teams, or projects for internal chargeback
  • Grouping — assign organizations, repositories, and users to cost centers
  • Visibility — enterprise owners see per-cost-center spending breakdowns
  • Default bucket — unassigned resources accumulate in the enterprise default cost center
Key limits: Up to 250 cost centers per enterprise, each supporting up to 25,000 resources.
Cost centers solve a fundamental enterprise problem: who pays for what? Without them, all GitHub spending lands on a single invoice with no departmental attribution. Finance teams need this for internal chargeback, budget accountability, and spend forecasting. The 250/25K limits are generous — most enterprises use 10–50 cost centers. Any resource not explicitly assigned goes to the default cost center, so always assign resources proactively.

Cost Center Design Patterns

Choose the model that fits your organization

Pattern 1: By Business Unit Pattern 2: By Project ┌──────────────┐ ┌──────────────┐ │ Engineering │ ← orgs + users │ Project-X │ ← repos + users │ Marketing │ │ Project-Y │ │ Data Science │ │ Platform │ └──────────────┘ └──────────────┘ Pattern 3: Hybrid ┌─────────────────────────────────────────┐ │ BU cost centers → license costs │ │ Project cost centers → usage costs │ │ (Actions, Packages, Codespaces) │ └─────────────────────────────────────────┘
  • By BU — simplest; one cost center per department, maps to org chart
  • By Project — aligns spend with initiatives; useful for project-based funding
  • Hybrid — BU for seats/licenses, project for consumption; most flexible
Pattern 1 (By BU) works best when departments own their own orgs and the org chart is stable. Pattern 2 (By Project) is ideal for companies that fund by initiative and need to track ROI per project. Pattern 3 (Hybrid) is the most common at large enterprises — license costs follow people (BU), while usage costs follow work (project). The hybrid approach requires more setup but gives the most accurate chargeback. Ask the customer: "How does your finance team allocate IT costs today?" — mirror that model.

Allocation Rules — What Goes Where

How each product's costs are attributed to cost centers

ProductAllocation ModelAttributed By
GHEC SeatsLicense-basedUser's assigned cost center
Copilot SeatsLicense-basedUser's assigned cost center
ActionsUsage-basedRepo/org where workflow runs
PackagesUsage-basedRepo/org that owns the package
CodespacesUsage-basedRepo/org where codespace is created
Secret Protection (GHAS)License-basedUser's assigned cost center
Code Security (GHAS)License-basedUser's assigned cost center
Copilot Premium RequestsUsage-basedUser's assigned cost center*

* Copilot Premium Requests are user-level consumption (premium models, agent mode) billed to the user's cost center, unlike other usage-based products which are attributed to the repo/org where the work runs.

⚠️ Understanding allocation rules is critical — misattribution is the #1 source of chargeback disputes.
This is the key slide. License-based costs follow the user — whoever the seat is assigned to, their cost center gets charged. This includes GHAS (Secret Protection and Code Security), which are license-based products allocated per user, the same way Copilot and GHEC seats work — each active committer is billed to one cost center based on their user assignment. Usage-based costs follow the resource — the org or repo where the compute happens determines the cost center. The exception is Copilot Premium Requests, which are usage-based but tied to the individual user's cost center since they represent per-user consumption. Walk through a concrete example: "Dev in Engineering cost center runs Actions in a Marketing repo — Actions cost goes to Marketing, Copilot seat and GHAS license costs stay with Engineering."

Tracking Copilot & GHAS Costs

Per-cost-center visibility for the biggest variable spend items

Copilot Tracking

  • Seat count per cost center in billing dashboard
  • Premium request consumption per cost center
  • Usage metrics: completions, chat, agent mode
  • Set budget alerts on premium request spend

GHAS Tracking

  • Active committers per cost center
  • Secret Protection & Code Security billed separately
  • Cross-repo committers may span multiple cost centers

Budget Alerts

TypeBehavior
Alert-onlyNotify at threshold; usage continues
Stop-usageBlock further usage at threshold (metered products only — not GHAS)
⚠️ Set budgets before rollout — retroactive budget enforcement does not refund overages.

Setup Order

  • 1. Create cost centers
  • 2. Set budgets per cost center
  • 3. Assign resources
  • 4. Then enable features
Emphasize the setup order — budgets should be configured BEFORE enabling Copilot or GHAS, not after. Alert-only budgets are good for visibility. Stop-usage budgets enforce hard caps but ONLY for metered products (Actions, Packages, Codespaces). GHAS and Copilot seat licenses are alert-only — stop-usage does not apply to license-based products. Start with alert-only at 80% of expected spend. Budget alerts apply at enterprise or cost center scope.

Azure Billing Integration

Unified billing through Azure Cost Management

How It Works

  • Link GitHub Enterprise to an Azure subscription
  • GitHub costs appear in Azure Cost Management
  • Cost centers map to Azure billing dimensions
  • Single invoice for GitHub + Azure resources

Per-Cost-Center Routing

  • Each cost center can route to a different Azure subscription or resource group
  • Enables departmental Azure billing alignment
  • Supports existing Azure cost allocation tags

Benefits

  • Single pane of glass — all cloud costs in one view
  • Existing workflows — leverage Azure cost alerts and budgets
  • Compliance — unified audit trail for procurement
  • Forecasting — Azure Cost Management projections include GitHub spend
Requires an Azure subscription with Owner or Contributor role to link.
This is critical for enterprises already on Azure — they get a single invoice and unified reporting across all cloud spend. The linkage happens at Enterprise Settings → Billing → Payment information. Once linked, GitHub metered usage (Actions, Packages, Codespaces, Copilot premium) flows through the Azure subscription. License costs (GHEC seats, Copilot seats) also appear in Azure Cost Management. Per-cost-center routing lets departments manage their own Azure subscriptions while the enterprise maintains centralized governance.

Cost center management can be fully automated. Common pattern: a scheduled GitHub Actions workflow runs monthly, pulls usage data per cost center via the API, generates a chargeback CSV, and emails it to department leads. For new resource assignment, use a webhook listener on `repository.created` and `organization.created` events to auto-assign to the correct cost center based on naming conventions. The `gh api` CLI command is the fastest way to explore these endpoints interactively.

Cost Centers Best Practices

Naming Conventions

  • dept-environment — e.g., engineering-prod
  • bu-team — e.g., platform-sre
  • Match your organization's GL codes where possible

Setup Checklist

  • ✅ Assign ALL resources — unassigned goes to default
  • ✅ Set budgets immediately after creation
  • ✅ Review quarterly and adjust with org changes
  • ✅ Document ownership: who approves spend per cost center?

Common Mistakes

  • ❌ Creating cost centers after features are enabled (retroactive = messy)
  • ❌ Leaving resources unassigned (inflates default cost center)
  • ❌ Too many cost centers (administrative overhead exceeds value)
  • ❌ No budget alerts (surprise invoices at month end)
  • ❌ Not reviewing when teams reorg (stale attribution)
⚠️ Start with fewer cost centers and split later — merging cost centers is harder than splitting.
Wrap up with actionable takeaways. The #1 mistake is creating cost centers after features are already in use — historical usage won't retroactively reassign. Start simple: one cost center per major department, budget alerts at 80%, and quarterly reviews. Naming should match whatever your finance team already uses for internal cost tracking. Document an owner for each cost center who can approve spend and adjust budgets. Start with fewer cost centers — you can always add more, but consolidating existing ones requires moving all resources manually. Transition to Day 1 wrap-up discussion and Q&A. Encourage participants to explore cost center settings in their own enterprise environments.
