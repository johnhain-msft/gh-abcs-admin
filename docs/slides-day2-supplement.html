Ruleset Anatomy

Targets
branches / tags → Rules
what's enforced → Bypass Actors
who can skip → Enforcement
Active · Evaluate · Disabled

Targeting patterns

  • Include: refs/heads/main, refs/heads/release/**
  • Exclude: hotfix/*
  • Default branch shortcut — applies regardless of name
  • Target by repository custom properties: environment = production

Evaluation Mode = Dry Run

Gathers compliance data via Rule Insights without blocking anyone — ideal for rollout

Organization vs Repository Rulesets

Organization Rulesets

  • Defined once at org level
  • Automatically cascade to all repos (incl. new ones)
  • Can target repos by custom properties
  • Only org owners can create / edit

Repository Rulesets

  • Scoped to a single repo
  • Repo admins can create / edit
  • Stack with org rulesets
  • Cannot weaken an org ruleset

Key principle  Multiple rulesets can apply — the most restrictive combination wins.

Key Ruleset Rules

  • Require pull requests — no direct push to target
  • Required reviews — 1, 2, or more approvals
  • Dismiss stale reviews — new push invalidates prior approval
  • Require CODEOWNERS review — owners must approve
  • Required status checks — CI must pass; lock to integration ID
  • Require signed commits — GPG, SSH, or S/MIME
  • Require linear history — squash or rebase only
  • Block force pushes — prevent history rewriting

+ Push rules  File path restrictions, file size limits, commit-message patterns, author email patterns

Tag Rulesets

Why protect tags?

  • Tags mark releases & milestones
  • Consumers pin to tags — reuse breaks reproducibility
  • Deletion erases release history

Common patterns

  • v* — all version tags
  • v[0-9].[0-9].* — strict semver
  • release-* — release candidates

Rules to enable

  • Restrict creation — only release managers
  • Block updates — tags are immutable after creation
  • Block deletion — require escalated approval

Bypass actor

Release engineering team with always bypass mode, automated via GitHub Actions

Code Scanning — Admin Enablement

What it does

CodeQL analyzes source code to find security vulnerabilities before they reach production.

How to enable

  • Security Configurations — apply at org level, repos inherit
  • Default Setup — zero-config; admin enables it, GitHub handles the rest

Cost

Part of GitHub Code Security — $30/active committer/month. Only committers to enabled repos count.

Monitoring

Track rollout and alert trends in the Security Overview dashboard (Coverage + Risk tabs).

Copilot Autofix  AI generates fix suggestions — reduces mean time to remediation and developer friction

Dependabot

Automated vulnerability alerts and dependency updates — free on all repos, no additional cost.

Three tiers

  • Alerts — passive notification when a dependency has a known CVE
  • Security updates — auto-opens PRs to patch vulnerable versions
  • Version updates — proactive PRs to keep dependencies fresh

Enable via Security Configurations at the org level. Dependabot is a cost-effective baseline — layer it with Code Security for deeper analysis.

Security Configurations

What is it?

Org-level policy bundles that define which security features are enabled and how

Options

  • GitHub Recommended — sensible defaults
  • Custom Config — pick features & settings

Includes toggles for

  • Code scanning (default / advanced)
  • Secret scanning + push protection
  • Dependabot alerts & updates
  • Dependency review enforcement
  • Private vulnerability reporting

Apply at scale: Select repos → Apply config → Done. New repos can auto-inherit.

  • Filter by severity, tool, repo, state

Key tabs

  • Risk — repos sorted by open alert count
  • Coverage — which repos lack which features

Campaigns

Active Committer Optimization

Key Optimization Techniques

  • Enable only on repos with active development — skip archived, read-only, and dormant repos
  • Unique user counting — one dev committing to 10 enabled repos = 1 committer
  • Bot accounts — GitHub App bots (Dependabot, Renovate) are excluded from billing; user-type automation accounts do count

The 90-Day Trailing Window

  • Committer stays active for 90 days after their last commit
  • Disabling a feature doesn't immediately reduce count
  • Plan for a lag period when projecting cost savings
  • Use the billing API to track active committer trends

Watch out  Archive dormant repos before enabling GHAS to avoid paying for inactive committers

Automation, gh CLI & Marketplace

Scripting admin at scale

gh CLI for Admin

Why gh?

  • Pre-authenticated on GitHub-hosted runners
  • Wraps REST and GraphQL APIs
  • Built-in --paginate for full result sets
  • --jq for inline JSON filtering
  • Extensions for custom admin commands

Key commands

gh api /orgs/{org}/repos --paginate --jq '.[]'

gh repo list my-org --json name,visibility

gh api /enterprises/{ent}/audit-log

Env vars: GH_TOKEN, GH_ENTERPRISE_TOKEN, GH_HOST

Admin Automation Patterns

Bulk team management

gh api orgs/{org}/teams --paginate
Add / remove members in bulk via loop

Audit log export

gh api /enterprises/{ent}/audit-log
  --method GET -F phrase='action:repo.create'
  --paginate

Dormant user reports

List members → check last activity via gh api → flag inactive accounts

Repo access audits

gh api repos/{owner}/{repo}/collaborators
  --paginate --jq '.[] | [.login, .role_name]'

Pattern: gh repo list → loop → gh api per repo → aggregate results

Commit Statuses & Check Runs

Status API (legacy)

  • States: pending · success · failure · error
  • External CI posts status to a commit SHA
  • Context string identifies the check

Check Runs API (modern)

  • Rich output: annotations, summaries, images
  • Actions: re-run, request changes
  • Scoped to a GitHub App installation

Tying it together with Rulesets

Required status checks reference these by context name. Best practice: lock to integration_id to prevent impersonation from similarly named checks.
